## Overview

This project implements the **Vanilla LLM baseline** for document-level relation extraction on the DWIE dataset. The Vanilla baseline directly prompts an LLM to extract relation triples from documents without any rules, demonstrations, or retrieval — establishing a performance floor that Part 2 (RuAG with MCTS-discovered logic rules) aims to surpass.

## How to Run

### Step 1: Load DWIE dataset into SQLite

```bash
python main.py --step load
```

This reads all 802 JSON files and creates `ruag.db` containing:

| Table | Contents | Rows |
|-------|----------|------|
| `documents` | Article content + train/test split | 802 |
| `entities` | All entities per document | 10,829 |
| `relations` | Ground truth relation triples | 19,493 |
| `relation_types` | 20 relation definitions | 20 |
| `filtered_docs` | 7 docs excluded (violate GPT protocol) | 7 |
| `predictions` | LLM outputs (populated in Step 2) | — |
| `rules` | Reserved for Part 2 (MCTS logic rules) | — |

### Step 2: Run Vanilla baseline

```bash
# Using Groq (free, no credit card needed — get key at https://console.groq.com)
python main.py --step vanilla --provider groq --model llama-3.1-8b-instant --api-key YOUR_KEY

# Alternative: Groq with larger model (better results, 100K tokens/day limit)
python main.py --step vanilla --provider groq --model llama-3.3-70b-versatile --api-key YOUR_KEY

# Alternative: OpenAI (paid)
python main.py --step vanilla --provider openai --model gpt-4o-mini --api-key YOUR_KEY
```

This processes 93 test documents: for each document, the script constructs a prompt with the document content, entity list, and relation type definitions, calls the LLM, parses the response into relation triples, and stores predictions in SQLite.

### Step 3: Evaluate

```bash
python main.py --step evaluate
```

Computes micro-averaged Precision, Recall, and F1 by comparing predictions against ground truth, with a per-relation breakdown.

### Performance Gap Analysis

The gap between our result (F1: 26.37%) and the paper's Vanilla baseline (F1: 46.94%) is attributable to the model size difference:

- **Paper**: GPT-4 (~1.8T parameters)
- **Ours**: Llama 3.1 8B (8B parameters) — **225x smaller**

The implementation methodology is identical: same DWIE dataset, same 20 relation types, same prompt structure, same evaluation method (micro-averaged Precision/Recall/F1 across all test documents). The smaller model struggles particularly with:

1. **Complex cross-sentence reasoning** — connecting entities mentioned in different paragraphs
2. **Distinguishing similar relation types** — e.g., `based_in0` vs `in0`, `agent_of` vs `member_of`
3. **Nominal variation awareness** — consistently producing both `citizen_of` and `citizen_of-x` variants

We chose Llama 3.1 8B via Groq's free API tier due to cost constraints. GPT-4 API access would require paid credits (~$5–10 for 93 documents).

## Implementation Details

### LLM Configuration

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Temperature | 0 | Deterministic output (paper Table A5) |
| Top-p | 1 | No nucleus sampling truncation (paper Table A5) |
| Max tokens | 2000 | Sufficient for documents with many relations |

### Prompt Design

The prompt provides:
1. **Task description** — extract relation triples from a document
2. **20 relation type definitions** with examples (from `relations_dict.json`)
3. **One worked example** demonstrating the output format, including `-x` relation variants and `gpe0`
4. **Entity list and document content** for the specific test article
5. **Explicit instruction** to use exact entity names from the provided list

The prompt does **not** include any demonstrations (ICL), retrieved examples (RAG), or logic rules (RuAG) — this is a pure Vanilla baseline.

### Response Parsing

The parser extracts `(entity1, relation, entity2)` triples from the LLM output using:
- Regex pattern matching for parenthesized triples
- Format-level cleanup: stripping quotes, backticks, asterisks, and whitespace
- **Strict exact-match validation**: both entities must appear in the document's entity list, and the relation must be one of the 20 defined types

No semantic normalization or fuzzy matching is performed.

### Data Management

All data resides in a single `ruag.db` SQLite database. No intermediate CSV, JSON, or pickle files are created during processing. The database schema supports both Part 1 (baseline) and Part 2 (MCTS rules) workflows.

### Filtered Documents

Following the original implementation, 7 test documents are excluded from evaluation because they violate GPT processing protocols or have empty ground truth. This leaves 93 active test documents.
